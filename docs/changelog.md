# CHANGELOG 自動生成（検討）

このドキュメントは、CHANGELOG を自動生成する仕組み（例: `conventional-changelog`）の導入可否を検討し、結論と根拠をまとめたものです。

## 背景

リリース時に「何が変わったか」をユーザーへ伝える手段として CHANGELOG は有用です。一方で、導入するとコミット運用（conventional commits 等）やリリース手順が固定化されます。

## 候補ツール

| ツール                   | 特徴                          | 向いているケース             | 注意点                             |
| ------------------------ | ----------------------------- | ---------------------------- | ---------------------------------- |
| `conventional-changelog` | conventional commits から生成 | 生成だけを軽量に足したい     | preset 選定や運用ルールが必要      |
| `standard-version`       | version bump + changelog      | まとめてリリース作業を自動化 | ワークフローがやや強めに規定される |
| `release-it`             | リリース全体の自動化          | 大規模/複数配布先            | 現状は過剰                         |
| `changesets`             | PR 単位で変更を管理           | 複数パッケージ/大規模        | 導入コストが高い                   |

## 導入判断

### メリット

- 形式が揃い、抜け漏れが減る
- 自動化しやすく、リリース作業が軽くなる

### デメリット

- コミットメッセージやリリース手順の規律が必要（移行期はノイズが増えやすい）
- 現状（beta/小規模）では、運用コストに見合わない可能性がある

## 結論

**今回は自動生成を導入しない（ドキュメントとして検討結果のみ残す）。**

根拠:

- 現在は `0.1.0-beta.0` の段階であり、仕様変更や改善が頻繁に起き得る
- 小規模であれば、CHANGELOG は当面手書きでも十分に回る
- 自動生成を入れる場合、conventional commits 前提の運用整備まで含めて着手する必要がある

## 当面の運用案（手動）

- リリースごとに `CHANGELOG.md`（リポジトリルート）を手動で更新
- 形式例:

```markdown
## [0.1.0] - YYYY-MM-DD

### Added

- ...

### Changed

- ...

### Fixed

- ...
```

## 再検討タイミング

- 1.0 に到達し、リリース頻度が上がった
- コントリビューターが増え、変更点の集約コストが上がった
- CI でリリース自動化を行う方針が固まった
