# ログ仕様とログローテーション（検討）

このドキュメントは、現在のログ出力仕様を踏まえて「ログローテーションが必要か」「必要なら導入できるか」を評価し、結論と根拠をまとめたものです。

## 現状のログ出力

### 出力先

1. `stderr`（主）: 進捗・エラー・状態メッセージを `console.error()` で出力
2. タスク別ログファイル: `$XDG_STATE_HOME/opencode/orchestrator/<task>/logs` に step/audit などのログを保存
3. autocommit の補助ログ: 状態ディレクトリ配下に JSONL を追記（デバッグ用途）

### ログの規模感（評価）

- ループ 1 ステップあたりの `stderr` 出力は概ね数行〜数十行
- `maxLoop=100` の最悪ケースでも、インタラクティブ用途として致命的な肥大化は起きにくい
- ただし、ステップごとのログファイル（`orch_step_*.txt`, `audit_step_*.jsonl` 等）はステップ数に比例して増える

## ログローテーションの必要性

現状の用途（開発・検証のための短期セッション中心）では、ログローテーションをアプリ内に実装しなくても運用可能と判断します。

根拠:

- タスク単位でログディレクトリが分離されており、不要になったタスク状態ごと削除できる
- ログは「サービス常駐」ではなく「ループ実行の補助情報」であり、長期蓄積を前提にしていない
- 依存追加（ログライブラリ）や出力仕様変更は、デバッグ性・互換性に影響し得る

## ライブラリ導入可否（比較）

| 候補                 | 使いどころ                            | メリット     | デメリット                         |
| -------------------- | ------------------------------------- | ------------ | ---------------------------------- |
| rotating-file-stream | ファイルローテーションのみ欲しい場合  | シンプル     | 運用設計が別途必要、現状は優先度低 |
| winston              | 多機能なロガー/トランスポート         | 機能豊富     | 依存が重め、設計変更が大きい       |
| pino (+ redact)      | 構造化ログ/機密情報対策も含めたい場合 | 高速・拡張性 | API 移行が必要、現状は過剰         |

## 結論

**今回はログローテーションを導入しない（次イテレーション以降に再検討）。**

再検討トリガー例:

- 長時間稼働（daemon 的）な運用が増え、ログが継続的に増加する
- ファイルベースの構造化ログを正式に導入する
- 複数人運用でログ保存・共有の要件が増える
