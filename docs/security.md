# セキュリティ考慮事項

## ログ出力における機密情報マスキング（検討）

このドキュメントは、ログ出力に含まれうる機密情報（パスワード・トークン等）を自動マスキングする仕組みについて、対象範囲・判断基準・実装方針の検討結果と結論をまとめたものです。

### 背景

オーケストレータは実行中に `stderr` へログを出し、タスクごとのログディレクトリにステップログ等を書き出します。ログにはプロンプト断片・コマンド出力・エラーメッセージが混在し得るため、誤って認証情報をログに残さない設計が望まれます。

### マスキング対象の判断基準

「認証情報や秘密鍵を含む文字列パターン」を対象とし、少なくとも以下を含みます。

- passwords（password/passwd など）
- tokens（token/bearer token など）
- API keys（api_key/api-key など）
- secrets（secret など）
- Authorization ヘッダー（`Authorization: Bearer ...`）

### 具体的な正規表現例（受け入れ条件に対応）

以下は検出候補の例です（最終的な実装では優先順位・誤検知/漏れの扱いを追加検討します）。

```text
/password[=:]\\s*[^\\s]+/i
/token[=:]\\s*[^\\s]+/i
/api[_-]?key[=:]\\s*[^\\s]+/i
/secret[=:]\\s*[^\\s]+/i
/authorization:\\s*Bearer\\s+[^\\s]+/i
```

### 実装アプローチの検討

1. 正規表現ベース置換（ログ出力文字列を置換してから出す）
   - メリット: 依存追加が不要で導入が軽い
   - デメリット: 呼び出し箇所が分散していると適用漏れが起きやすい

2. 専用ラッパー関数（`console.error` を直接呼ばず、`logError()` のような関数に集約）
   - メリット: 適用点が集約され、段階的な改善（構造化ログ/フィルタ）も可能
   - デメリット: 既存コードの置換が広範囲になりやすく、影響範囲が大きい

3. 専用ライブラリ（例: pino の redaction / pino-redact 等）
   - メリット: 既存知見があり、機能が豊富
   - デメリット: 依存サイズ増・移行コストが発生

#### 根拠

- 影響範囲が広く（ログ出力が多数箇所に分散）、安全な導入には追加テストと設計の固定が必要
- 誤検知/漏れのリスクがあり、拙速に入れるとデバッグ性低下や期待しないマスキングを招く
- 今回のリリース準備では「機密ファイルのコミット防止」等の直接的な安全策を優先する
